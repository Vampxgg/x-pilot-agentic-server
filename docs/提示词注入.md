# Document Generation 提示词注入说明

本文档说明 `document-generation` 智能体调用时，**系统提示词（System Prompt）** 和 **用户消息（Human Message）** 的组成、来源及注入流程。对于多 Agent 编排，各子 Agent 也适用类似的注入规则。

---

## 一、整体结构

```
LLM 收到的 messages:
├── SystemMessage(systemPrompt)   ← buildSystemPrompt 组装
└── HumanMessage(instruction)     ← buildInstruction 组装
```

---

## 二、System Prompt 组成

由 `src/core/agent-graph.ts` 的 `buildSystemPrompt()` 按顺序拼接，各块之间用 `\n\n---\n\n` 分隔。

| 顺序 | 区块 | 来源 | 说明 |
|------|------|------|------|
| 1 | `# Identity` | `IDENTITY.md` | 角色、能力 |
| 2 | `# Soul` | `SOUL.md` | 价值观、硬约束、决策启发 |
| 3 | `# Mission` | `MISSION.md` | 目标、工作流、图片/表格策略 |
| 4 | `# Tool Usage Guidelines` | `TOOLS.md` | 工具使用说明 |
| 5 | 其他 .md | `HEARTBEAT.md` 等 | 不在 identity/soul/mission/tools 中的 prompts |
| 6 | Skills | `skills/*.md` | 动态技能（若有） |
| 7 | `# Task Context` | 业务路由传入的 `context` | 本次任务业务参数 |
| 8 | `# Session` | `sessionId` | 工作区会话标识 |
| 9 | `# Long-term Memory` | `memory/MEMORY.md` | 长期记忆 |

### Task Context 示例（基于测试请求）

```json
{
  "businessType": "document-generation",
  "businessInput": { "smart_search": true, "database_id": "" },
  "topic": "智能网联汽车视觉传感器",
  "templateName": null,
  "templateType": null
}
```

## 三、Human Message（Instruction）组成

由 `apps/document-generation/code/routes.ts` 的 `buildInstruction()` 生成，作为单条 HumanMessage 的 content。

### 结构模板

```
你是教学资源生成专家。{有模板/无模板 分支说明}

【核心规则】
{structureRule}
4) 优先使用已提供知识与 references；不足时调用 knowledge_search 补充。{datasetHint}
{largeTextHint}
5) 图片：...（按骨架/自由插入 分支）
6) 或 7) 仅输出最终 Markdown 正文...

【结构化输入(JSON)】
{payload 含 select_knowledge_unit、prompt、large_text、smart_search 等}

{有模板时：【模板结构骨架】+【模板原文】}
```

### 有模板 vs 无模板

| 条件 | `hasTemplateContent` | 结构规则 | 图片规则 |
|------|----------------------|----------|----------|
| 有 template 正文或骨架章节 >1 | true | 严格按骨架章节顺序生成 | 仅按骨架 image 位置生成 |
| docTemplate 为空、无骨架 | false | 自由规划章节 | 按教学逻辑插入，但 rules 5 仍限制「勿超额」 |

## 四、测试请求示例下的实际注入内容

### 请求体摘要

```json
{
  "select_knowledge_unit": "智能网联汽车视觉传感器",
  "prompt": "至少需要5张配图",
  "database_id": "",
  "smart_search": true,
  "large_text": "**模块2：单传感器装调技术**\n\n...(约5万字)",
  "docTemplate": "",
  "is_single_file_processing": true,
  "sessionId": "vampxgg-5"
}
```

### System Prompt 涉及的关键约束

- **SOUL**：未提供 database_id 时禁止 knowledge_search；smart_search=true 时允许联网。
- **MISSION Image Strategy**：仅当骨架标注 image 时才插入图片；「用户 prompt 中的『图片 N 张』为参考期望，**不得为满足数量而超出骨架范围额外生成**」。
- **TOOLS image_generate**：仅在对应配置且合适网络图片缺失时调用（此工具在目前的多Agent流程中已移除到外层统一处理）。

### Human Message 实际内容（摘要）

```
你是教学资源生成专家。请根据主题生成教学 Markdown 文稿。

【核心规则】
1) 无模板，请根据主题「智能网联汽车视觉传感器」自由规划...
2) 内容保持教学可执行，可结合 prompt 中的补充要求。
3) 按教学逻辑安排文本、表格、列表等...
4) 【警告】未提供 database_id，绝对禁止调用 knowledge_search...
【数据优先级】large_text（用户上传内容）为最高优先级...
5) 图片：仅按骨架标注的 image 位置生成...勿为满足 prompt 中的「图片 N 张」而超额生成...
6) 仅输出最终 Markdown 正文...

【结构化输入(JSON)】
{
  "select_knowledge_unit": "智能网联汽车视觉传感器",
  "prompt": "至少需要5张配图",
  "database_id": null,
  ...
  "large_text": "**模块2：单传感器装调技术**\n\n...(完整约5万字)"
}
```

## 五、注入流程

```
请求 body
    │
    ▼
normalizeRequest(body) → norm
    │
    ├─────────────────────────────────────────┐
    │                                         │
    ▼                                         ▼
resolveTemplateContent(norm.template)    buildInstruction(norm, skeleton, excerpt, ...)
    │                                         │
    └─────────────────────────────────────────┘
                    │
                    ▼
        instruction (Human Message content)
                    │
                    ▼
streamAgent(agentName, instruction, opts)
    │
    ├─► buildSystemPrompt(agentDef, longTermMemory, taskContext, sessionId)
    │       │
    │       └─► systemPrompt (System Message content)
    │
    └─► compiled.stream({
          messages: [new HumanMessage(instruction)],
          systemPrompt,
          ...
        })
            │
            ▼
    createThinkNode: model.invoke([
      SystemMessage(systemPrompt),
      ...workingMemory,
      ...state.messages  // 含 HumanMessage(instruction)
    ])
```

## 六、已知冲突与改进方向

### 冲突：用户要求「至少5张配图」 vs 系统约束

| 用户期望 | 系统约束 |
|----------|----------|
| prompt: "至少需要5张配图" | MISSION: 仅骨架标注 image 才插入；不得为满足数量超额生成 |
| 无模板场景下也需要配图 | 无模板时 skeleton 无 image，规则禁止主动插图 |

### 改进建议

1. **MISSION.md Image Strategy**：增加「无模板场景」分支，明确：当 `templateProvided=false` 且用户 prompt 要求配图时，应按教学逻辑在合适位置插入图片。
2. **buildInstruction**：在 instruction 开头增加 `【用户明确要求】\n${prompt}\n`，突出用户要求。
3. **规则 5 表述**：区分有模板/无模板，无模板时改为「若用户 prompt 要求配图数量，请按教学逻辑合理插入」。
